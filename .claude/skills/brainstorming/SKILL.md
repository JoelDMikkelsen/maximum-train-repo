# Brainstorming

## The Rule

**Do NOT write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it.**

## The Process

### Step 1: Explore Context

- Read existing files (SPEC.md, TASKS.md, any docs/)
- Review recent changes (git log)
- Understand what exists already

### Step 2: Ask Clarifying Questions

- One question at a time
- Multiple-choice when possible
- Understand purpose, constraints, success measures

### Step 3: Propose Alternatives

- Present 2-3 different approaches
- Include trade-off analysis
- Let user choose direction

### Step 4: Present the Design

- Share design in digestible sections
- Get approval for each section before proceeding
- Scale detail to complexity

### Step 5: Document the Design

- Save to `docs/plans/YYYY-MM-DD-<topic>-design.md`
- Commit the design document

### Step 6: Create Implementation Plan

- Invoke the writing-plans skill
- No other implementation skills yet

## Key Principles

- Single questions only - don't overwhelm
- Multiple-choice preferred for easier responses
- Incremental validation - approve each section
- Challenge assumptions - even "simple" projects need design review
- Return to clarification when gaps emerge

## Terminal State

The only next step after brainstorming is writing-plans. No implementation until plan is written and approved.
