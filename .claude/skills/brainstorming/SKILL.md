# Brainstorming

## The Rule

**Do NOT write code until you have a clear direction and user has approved it.**

## Fast-Track Path (when SPEC.md + TASKS.md exist)

This project already has detailed specification and task documents. When these exist:

1. Read SPEC.md and TASKS.md
2. Summarize your understanding briefly
3. Flag any ambiguities or questions (keep it short)
4. If user says "go" or equivalent, proceed directly to writing-plans
5. Don't force lengthy discovery for requirements that are already documented

## Full Path (new features without existing spec)

### Step 1: Explore Context
- Read existing files, docs, recent changes

### Step 2: Ask Clarifying Questions
- One question at a time, multiple-choice when possible

### Step 3: Propose Alternatives
- Present 2-3 approaches with trade-offs

### Step 4: Present the Design
- Share in digestible sections, get approval per section

### Step 5: Document the Design
- Save to `docs/plans/YYYY-MM-DD-<topic>-design.md`

### Step 6: Create Implementation Plan
- Move to writing-plans skill

## Key Principles

- Don't over-process when specs already exist
- Single questions only - don't overwhelm
- Scale process to complexity
- Return to clarification when gaps emerge
